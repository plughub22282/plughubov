# Онбординг новых пользователей — дизайн

Дата: 2026-07-09
Статус: спроектировано, ожидает `writing-plans`
Часть более широкой инициативы «маркетинг в приложении» (подпроект 1 из 3: онбординг → промо в приложении → виральность/рефералка).

## Контекст

Сейчас после входа через Discord (`AuthScreen.tsx`) пользователь сразу попадает в главный интерфейс (`App.tsx` → `Home.tsx`) без какого-либо промежуточного шага. Ни признака «новый пользователь», ни флага «онбординг пройден» в системе нет ни на клиенте, ни на сервере.

Цель — ввести одноразовый экран ценности + короткая анкета (DAW, жанр) сразу после первого входа, плюс необязательный coach-marks тур по интерфейсу, доступный по кнопке в конце анкеты.

## Цели онбординга

1. Объяснить ценность продукта (2-3 пункта: что за приложение и зачем).
2. Собрать данные для персонализации: DAW и музыкальный жанр пользователя.
3. Дать возможность (не обязательную) пройти короткий тур по ключевым точкам интерфейса.

Онбординг обязателен к прохождению целиком (без общего пропуска «в приложение»), но отдельные вопросы анкеты (DAW/жанр) можно пропустить по отдельности.

## Поток

Состояние — простой автомат внутри нового компонента `Onboarding.tsx`, шаги:

```
'value' → 'daw' → 'genre' → 'done'
```

- **`value`** — статичная информационная карточка: заголовок, 3 пункта ценности с иконками, кнопка «Далее». Прогресс-точки (3 шт.) отображают value → daw → genre.
- **`daw`** — вопрос «Какую DAW используешь?», чипы-варианты (FL Studio, Ableton, Logic Pro, FL Studio Mobile, Другое), одиночный выбор — клик по варианту сразу сохраняет ответ и переводит на следующий шаг. Кнопка «Пропустить» рядом с прогресс-точками переводит на `genre` без сохранения ответа.
- **`genre`** — аналогично: чипы жанров (Trap, House, Lo-fi, Drill, Pop, Другое), автопереход на `done`. «Пропустить» — на `done` без сохранения.
- **`done`** — «Готово! Показать быстрый тур по интерфейсу?» + две кнопки: `«Показать тур»` (primary) и `«Начать пользоваться»` (secondary).

При достижении `done` (любым путём — с ответами или без) состояние сохраняется в БД: `onboarding_completed = true`, а также `onboarding_daw`/`onboarding_genre`, если они были заданы (иначе остаются `null`).

Coach-marks тур (`OnboardingTour.tsx`) запускается только по явному клику «Показать тур» на шаге `done`. Тур ничего не пишет в БД — если пользователь его не запустил, повторного доступа к нему нет (ни из Settings, ни откуда-либо ещё в этом цикле).

## Данные

### Supabase (`supabase/schema.sql`)

Новые колонки в `public.profiles`, добавляются идемпотентно (аналогично `premium_until`):

```sql
alter table public.profiles add column if not exists onboarding_completed boolean not null default false;
alter table public.profiles add column if not exists onboarding_daw text;
alter table public.profiles add column if not exists onboarding_genre text;
```

`default false` — намеренно и распространяется на уже существующих пользователей: при следующем входе все пользователи (включая существующих) один раз увидят онбординг. Это сознательный побочный эффект: он нужен, чтобы собрать DAW/жанр и с текущей базы, а не только с новых регистраций.

Эти три поля — обычные пользовательские настройки, не роль/премиум/рефералка, поэтому `prevent_role_change`-триггер их не защищает: пользователь обновляет их обычным `update` через существующую RLS-политику `profiles_update_own`.

### Main-процесс (`src/main/auth.ts`)

- `AuthState` (и зеркальный тип в `src/renderer/src/types.ts`) пополняется полем `onboardingCompleted: boolean`.
- Читается вместе с `premium_until` в существующем `fetchDbPremiumUntil`-подобном запросе к `profiles` внутри `buildState()` — один общий SELECT, без лишнего похода в БД.
- Если `onboarding_completed` не пришёл вовремя (например, гонка с триггером `handle_new_user`, создающим профиль) — трактуется как `false` (показываем онбординг; лучше лишний раз показать, чем никогда).
- Новый IPC-метод `auth:completeOnboarding(daw: string | null, genre: string | null): Promise<AuthResult>` — апдейтит `profiles` (`onboarding_completed = true`, плюс переданные `daw`/`genre`, не трогая поле, если пришёл `null`) и возвращает свежий `AuthState`, по аналогии с `redeemPremium`.
- Экспонируется в `preload.ts` и `window.api.auth.completeOnboarding` — взаимодействие Renderer ↔ Main только через этот путь, без прямых обращений к Supabase из renderer (соответствует правилу CLAUDE.md о строгом разделении Main/Renderer).

## Компоненты

### `src/renderer/src/components/Onboarding.tsx`

Экспортирует `<Onboarding onDone={...} onStartTour={...} />`.

- Полноэкранный оверлей `fixed inset-0 z-[60]`, полупрозрачный затемняющий фон (`backdrop-blur` + `bg-black/50`) поверх основного UI (сайдбар + Home уже смонтированы под ним) — оверлей, а не блокирующий полноэкранный роут.
- Визуальный язык — как `AuthScreen.tsx`: `.card`, `animate-slide-up`, ambient glow, тот же градиентный акцент. Карточка шире, чем в `AuthScreen` (360px вместо 300px) — на шагах анкеты нужна сетка из 3-4 кнопок-чипов в ряд.
- Рендерится в `App.tsx` условно: когда `auth.status === 'signedIn' && !auth.state.onboardingCompleted`.
- Иконки для пунктов ценности переиспользуют существующий svg-паттерн из `App.tsx` (стиль `IconCatalog`/`IconMarket`/`IconReferral`), не выносятся в общий модуль — остаются локальными в `Onboarding.tsx`, по аналогии с тем, как иконки уже дублируются как локальные константы в `App.tsx`.

### `src/renderer/src/components/OnboardingTour.tsx`

Экспортирует `<OnboardingTour onFinish={...} />`.

- 4 точки подсветки: разделы сайдбара, глобальный поиск, мини-плеер, Premium CTA в сайдбаре — помечаются атрибутом `data-tour="sidebar-sections" | "search" | "player" | "premium-cta"` на уже существующей разметке `App.tsx`.
- Если целевой элемент с нужным `data-tour` не смонтирован на момент показа этого шага (например, `data-tour="premium-cta"` отсутствует у пользователя с активным премиумом, т.к. CTA рендерится только `!isPremium`) — этот шаг тура пропускается автоматически, без ошибки.
- Подсветка через `getBoundingClientRect()` целевого элемента: полупрозрачная маска с вырезом (spotlight) + всплывающая подсказка со стрелкой и текстом. Кнопки «Далее» / «Пропустить тур» / «Готово» (на последнем шаге).
- Координаты пересчитываются при `resize`; если конкретно не успевает — тур просто закрывается при `resize` (без усложнения логики пересчёта на лету — окно Electron в первые секунды использования обычно не ресайзят).
- `App.tsx` держит `showTour: boolean` state, включается колбэком `onStartTour` из `Onboarding`. Тур ничего не пишет в БД.

### `src/renderer/src/i18n.tsx`

Новые ключи namespace `onboarding.*` (`onboarding.welcome.title`, `onboarding.value.point1..3`, `onboarding.daw.question`, `onboarding.daw.option.*`, `onboarding.genre.question`, `onboarding.genre.option.*`, `onboarding.done.title`, `onboarding.done.startTour`, `onboarding.done.start`, `onboarding.skip`, `onboarding.tour.sidebarSections`, `onboarding.tour.search`, `onboarding.tour.player`, `onboarding.tour.premiumCta`, `onboarding.tour.next`, `onboarding.tour.skip`, `onboarding.tour.finish`, `onboarding.error.saveFailed`, `onboarding.error.retry`) для ru и en, по существующему паттерну `t('namespace.key', { vars })`.

## Персонализация Home по жанру

`Home.tsx` получает жанр пользователя (через `useAuth()` или проп из `App.tsx`) и применяет soft-фильтр к секциям «Для вдохновения» и «В тренде»: элементам, чьи `tags` содержат выбранный жанр, отдаётся приоритет; если подходящих меньше разумного минимума — довыбираются остальные элементы, чтобы секция никогда не оказалась пустой из-за фильтра.

DAW **не используется** для фильтрации контента в этом цикле — в каталоге сейчас нет поля DAW ни у плагинов (`Plugin`), ни у ассетов (`CommunityPlugin`), только `category`/`tags`. Ответ на вопрос про DAW сохраняется в `profiles.onboarding_daw` только для будущей аналитики/фич, реальной фильтрации Home в этом цикле не выполняет.

## Обработка ошибок и граничные случаи

- **Сбой сохранения** (`auth:completeOnboarding` вернул `ok: false`, например нет сети): оверлей не закрывается — инлайн-ошибка в карточке (паттерн `error && <div className="...status-error...">`, как в `AuthScreen.tsx`) с кнопкой повторить попытку.
- **`onboardingCompleted` не определён** (гонка с триггером создания профиля): трактуется как `false` — онбординг показывается.
- **Разрыв соединения посреди тура**: тур ничего не пишет в БД, просто закрывается локально без побочных эффектов.
- **Resize окна во время тура**: тур закрывается (без пересчёта позиций на лету).
- **Пропуск шагов DAW/жанра**: соответствующее поле остаётся `null`, `onboarding_completed` всё равно ставится `true` при достижении `done`. Жанр-фильтр на Home в этом случае не активен — обычная нефильтрованная лента.
- **Существующие пользователи**: увидят онбординг один раз при следующем входе (см. раздел «Данные» — намеренно).
- **Пользователь с активным премиумом**: шаг тура `premium-cta` автоматически пропускается (CTA не смонтирован в DOM).

## Вне рамок этого цикла

- Промо-баннеры внутри приложения (Home/Marketplace) — отдельный будущий цикл дизайна.
- Изменения в реферальной механике — отдельный будущий цикл дизайна.
- DAW не становится полем фильтрации каталога — только хранится для будущей аналитики.
- Повторный запуск coach-marks тура (например, из Settings) — не реализуется в этом цикле.

## Проверка при реализации

- `npx tsc --noEmit` и `npm run build` — чистый прогон, без новых warning.
- Ручной чек-лист в dev-сборке:
  - Первый вход нового Discord-аккаунта → оверлей ценности → DAW → жанр → done → (опционально) тур со всеми доступными точками подсветки.
  - Повторный вход после `onboarding_completed = true` → оверлей не показывается.
  - «Пропустить» на шаге DAW и/или жанра → поток не ломается, `onboarding_completed` всё равно ставится `true` на `done`.
  - Отключённая сеть в момент завершения → показывается ошибка с кнопкой повтора, оверлей не закрывается сам.
  - Пользователь с активным премиумом → шаг тура `premium-cta` не показывается / корректно пропускается.
- SQL-миграция в `supabase/schema.sql` идемпотентна (`add column if not exists`), соответствует стилю остального файла.
